**Step A — Widen `SearchResult`** (additive, non-breaking):

Append five optional nullable fields to the existing `SearchResult` interface. Keep the original six fields in their current order; add the new fields AFTER `address`:

```ts
export interface SearchResult {
  title: string;
  url: string;
  snippet: string;
  price: string | null;
  rating: number | null;
  reviewCount: number | null;
  address: string | null;
  // --- Phase 53: restaurant-enrichment fields (nullable; undefined on non-restaurant queries) ---
  photoUrl?: string | null;      // direct image URL; WhatsApp auto-unfurls as link preview
  openNow?: boolean | null;      // true=open, false=closed, null=unknown
  priceLevel?: string | null;    // "$" | "$$" | "$$$" | "$$$$" — pass-through token, no normalization
  cuisine?: string | null;       // free-text tag (e.g. "Italian", "פיצה", "Pan-Asian")
  reservationUrl?: string | null; // OpenTable / TheFork / venue's own booking URL
}
```

**Critical constraints:**
- Fields are BOTH `?:` (optional at compile time) AND `| null` (nullable at runtime). This matches the v1.4 convention (`rating: number | null`) while staying source-compatible with the three existing places `SearchResult` objects are constructed (`geminiMapsSearch` result.slice line 90, `geminiGroundedSearch` result.slice line 216, `knowledgeFallback` line 329) WITHOUT having to touch their field lists. An existing caller reads `result.photoUrl` → `undefined`; new code reads `result.photoUrl ?? null` → `null`. No TS errors, no behavior change for hotels/activities/general paths.

**Field extraction — restaurant branch:**

Replace the `results` map (current lines 90-98) with a branch-aware extraction. The non-restaurant path stays byte-identical; the restaurant path adds the five new fields:

```ts
const results: SearchResult[] = parsed.slice(0, resultCount).map((item: Record<string, unknown>) => {
  const base: SearchResult = {
    title: String(item.title ?? 'Result'),
    url: String(item.url ?? ''),
    snippet: '',
    price: null,
    rating: typeof item.rating === 'number' ? item.rating : null,
    reviewCount: typeof item.reviewCount === 'number' ? item.reviewCount : null,
    address: typeof item.address === 'string' ? item.address : null,
  };

  if (isRestaurantQuery) {
    return {
      ...base,
      photoUrl: typeof item.photo_url === 'string' ? item.photo_url : null,
      openNow: typeof item.open_now === 'boolean' ? item.open_now : null,
      priceLevel: typeof item.price_level === 'string' ? item.price_level : null,
      cuisine: typeof item.cuisine === 'string' ? item.cuisine : null,
      reservationUrl: typeof item.reservation_url === 'string' ? item.reservation_url : null,
    };
  }

  return base;
});
```

Note the key casing bridge: Gemini returns snake_case (`photo_url`, `open_now`, `price_level`, `cuisine`, `reservation_url`) — the wire shape we asked for in the prompt — but TS-land stays camelCase (`photoUrl`, `openNow`, etc.) matching the existing `reviewCount` convention in `SearchResult`.
