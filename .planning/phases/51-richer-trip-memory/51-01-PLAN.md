<objective>
Add v2.1 richer-trip-memory columns to `trip_decisions` and `trip_contexts`, create new `trip_archive` table, and provide typed query helpers all downstream plans (classifier, self-report, conflict detector, archive cron) will use.

Purpose: foundation layer — every other Phase 51 plan reads/writes these columns. Landing this first means Wave 2 plans can develop in parallel against a stable schema.

Output: Drizzle migration `0022_v21_phase51_trip_memory.sql`, updated `src/db/schema.ts`, extended `src/db/queries/tripMemory.ts`, and one decision documented in the plan: **decision archival = boolean `archived` flag on `trip_decisions`** (not FK, not sibling table). See `<decision_rationale>` below.
</objective>

<decision_rationale>
**Decision-archival strategy: boolean `archived` flag on `trip_decisions`.**

Options considered:
- **FK to trip_archive.id**: requires every decision row to point to an archive row, but decisions belong to an active trip_context (group_jid) whose identity moves to trip_archive under a new UUID — adding a nullable FK doubles update cost (move context, then rewrite every decision's FK) with no query benefit.
- **Sibling `trip_decisions_archive` table**: clean separation but duplicates 10+ columns, complicates dashboard queries (Phase 55) that want unified history, forces every read site to UNION.
- **Boolean `archived` flag (CHOSEN)**: single column add, archive cron flips it in one UPDATE, all existing `WHERE group_jid = ?` queries get a trivial `AND archived = 0` filter. Dashboard can opt in to archived history with a single boolean toggle. No FK migrations.

Trade-off: rows accumulate in `trip_decisions` forever. Acceptable — decisions are small (<1KB/row), one active trip per group, so even a heavy user generates <10K rows lifetime.
</decision_rationale>
