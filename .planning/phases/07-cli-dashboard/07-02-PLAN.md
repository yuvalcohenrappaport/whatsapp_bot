1. Create `cli/commands/contacts.ts`:
   - Export `addContactsCommand(program: Command): void`
   - Create `contacts` parent command with 4 subcommands:

   **`contacts list`:**
   - Query all contacts from DB: `db.select().from(contacts).all()`
   - For each contact, get latest inbound message (fromMe=false) timestamp + body preview (truncated to 40 chars)
   - Use `renderToString` with the Table component from `../ui/Table.js`. Headers: JID, Name, Mode, Last Message. Widths: [30, 18, 8, 44].
   - Truncate JID to show just the phone number part (strip `@s.whatsapp.net`)
   - Color the mode column: green for "auto", yellow for "draft", gray/dim for "off"
   - Print with `process.stdout.write(output + '\n')`

   **`contacts add <jid>`:**
   - Options: `-n, --name <name>` (display name), `-m, --mode <mode>` (off|draft|auto, default "draft")
   - Normalize JID: if it doesn't contain `@`, append `@s.whatsapp.net`
   - Use existing query pattern: `db.insert(contacts).values({ jid, name, mode }).onConflictDoNothing().run()` — but since `upsertContact` from queries/contacts.ts imports from `src/db/client.ts` (which imports src/config.ts), do NOT import the existing query functions. Instead, use `db` from `../db.js` and write the query inline using the schema tables.
   - After insert, also set the mode: `db.update(contacts).set({ mode }).where(eq(contacts.jid, jid)).run()`
   - Print success: `Added contact: <jid> (mode: <mode>)`

   **`contacts remove <jid>`:**
   - Normalize JID as above
   - Soft-delete: `db.update(contacts).set({ mode: 'off', updatedAt: Date.now() }).where(eq(contacts.jid, jid)).run()`
   - Print: `Removed contact: <jid> (set to off)`

   **`contacts configure <jid>`:**
   - Options: `-m, --mode <mode>`, `-r, --relationship <text>`, `-i, --instructions <text>`
   - Normalize JID
   - Build update object from provided options only (skip undefined)
   - Always add `updatedAt: Date.now()` to the set
   - `db.update(contacts).set(updates).where(eq(contacts.jid, jid)).run()`
   - Print: `Updated contact: <jid>` and list what was changed

   IMPORTANT: All DB access uses `db` from `../db.js` (NOT from `../../src/db/client.js`). Import schema tables from `../../src/db/schema.js`. This avoids the config.ts Zod validation issue.

1. Create `cli/commands/import.ts`:
   - Export `addImportCommand(program: Command): void`
   - Register: `program.command('import <file>').description('Import WhatsApp .txt chat history').requiredOption('--contact <jid>', 'Contact JID')`

   The action handler:
   a. Resolve the input file path to absolute (relative to cwd).
   b. Verify the file exists: `await fs.access(filePath)`. If not, print error and `process.exit(1)`.
   c. Normalize the contact JID: if it doesn't contain `@`, append `@s.whatsapp.net`.
   d. Derive the import filename from the JID. The existing `importChats()` expects files named `<phone>.txt` where phone becomes JID. So extract the phone from the JID (everything before `@`), and the dest filename is `<phone>.txt`. If the JID already contains `@`, use `jid.split('@')[0] + '.txt'`.
   e. Read config values for import/processed dirs: use env vars with defaults matching src/config.ts defaults:
      - `IMPORT_DIR` → `process.env.IMPORT_DIR ?? './data/imports'` (resolve relative to project root)
      - `PROCESSED_DIR` → `process.env.PROCESSED_DIR ?? './data/processed'` (resolve relative to project root)
      - `OWNER_EXPORT_NAME` → `process.env.OWNER_EXPORT_NAME` — this is REQUIRED. If missing, print error: "OWNER_EXPORT_NAME env var is required for import (your name as it appears in WhatsApp exports)" and exit 1.
   f. Ensure import dir exists: `await fs.mkdir(importDir, { recursive: true })`.
   g. Copy the file to the import dir with the JID-derived filename: `await fs.copyFile(filePath, path.join(importDir, destFilename))`.
   h. Call `importChats(importDir, processedDir, ownerExportName)`.

   CRITICAL: `importChats` imports from `src/db/client.ts` (which imports `src/config.ts`). This will fail without all env vars. There are two options:
   - Option A: The user must have the .env file sourced (PM2 does this automatically; the shell alias should too). Add a note to check for this.
   - Option B: Before calling importChats, we need GEMINI_API_KEY in env (for style summary generation).

   Since import calls `generateStyleSummary` which needs `GEMINI_API_KEY`, and the user is running from SSH where PM2's env may not be loaded, the import command MUST load the .env file first. Use `dotenv` (already a dependency) at the top of the import command: `import { config as dotenvConfig } from 'dotenv'` and call `dotenvConfig({ path: path.join(PROJECT_ROOT, '.env') })` BEFORE importing from `src/importer/importChats.js`.

   Actually, because of ESM static imports, we cannot call dotenvConfig before imports. Instead: use dynamic import. The import command should:
   1. Call `dotenvConfig({ path: ... })` to load .env
   2. Then `const { importChats } = await import('../../src/importer/importChats.js')` (dynamic import)

   This ensures the .env is loaded before src/config.ts evaluates.

   i. Use `render` + `waitUntilExit` with an Ink component that shows progress: "Importing <filename> for contact <jid>..." then "Import complete!" when done. The component fetches import status via useEffect, calls exit() when done.

   j. After import completes, call `process.exit(0)` after `waitUntilExit()` to ensure clean exit (importChats may leave handles open from Gemini).
