1. Create `cli/commands/drafts.ts`:
   - Export `addDraftsCommand(program: Command): void`
   - Create `drafts` parent command with 3 subcommands:

   **`drafts list`:**
   - Query all pending drafts from DB: `db.select().from(drafts).where(eq(drafts.status, 'pending')).orderBy(desc(drafts.createdAt)).all()`
   - For each draft, join contact name: `db.select({ name: contacts.name }).from(contacts).where(eq(contacts.jid, draft.contactJid)).get()`
   - For each draft, join the inbound message body: `db.select({ body: messages.body }).from(messages).where(eq(messages.id, draft.inReplyToMessageId)).get()`
   - Render with Table component (from `../ui/Table.js`). Headers: ID (first 8 chars), Contact, Their Message, Draft Reply, Created.
     - Widths: [10, 18, 30, 30, 18]
     - Truncate "Their Message" and "Draft Reply" to fit column width (use substring + "...")
     - Format "Created" using formatDate helper
   - If no pending drafts, print: "No pending drafts"
   - Print with renderToString + process.stdout.write

   **`drafts approve <id>`:**
   - NOTE: The CLI cannot SEND the message via WhatsApp because it does not have access to the WASocket (that lives in the bot process only). The CLI approve command sets the draft status to 'sent' in the DB. The actual message sending happens through the web dashboard or bot process. However, since the user is managing via CLI, this is effectively "marking as approved/sent" in the database. Print a clear note about this.
   - Actually, re-examining: the web dashboard's approve endpoint calls `sock.sendMessage()` via shared state. The CLI has no sock. So the CLI approve should:
     a. Find the draft and verify it's pending
     b. Mark it as 'sent': `db.update(drafts).set({ status: 'sent', actionedAt: Date.now() }).where(and(eq(drafts.id, id), eq(drafts.status, 'pending'))).run()`
     c. Print: `Draft <id> approved (marked as sent). Note: message was marked in DB but not sent via WhatsApp. Use the web dashboard to send messages directly.`

   WAIT — on reflection, this is a significant UX issue. The user expects "approve" to actually send the message. Let me reconsider. The CLI could hit the Fastify API's approve endpoint (POST to localhost:3000/api/drafts/:id/approve) which has access to the sock. But the research says "no HTTP dependency."

   Resolution: The CLI drafts approve command will update the DB status only. Print a warning: "Draft approved in DB. The bot process will NOT auto-send approved drafts — use the web dashboard's approve button to send directly via WhatsApp." This is the honest approach. The CLI is for database management, not for triggering WhatsApp sends. The user can also re-architect later if needed.

   Actually, a better approach: allow the approve command to optionally hit the local API if the bot is running. Add `--send` flag. Default behavior (no flag) just marks in DB. With `--send`, POST to `http://localhost:${API_PORT}/api/drafts/${id}/approve` with the draft body. Read API_PORT from env (default 3000). For auth, read JWT_SECRET from env and sign a token, OR read DASHBOARD_PASSWORD and call /api/auth/login first. This is getting complex.

   SIMPLEST approach per research guidance ("no HTTP"): Just mark the status in DB. Print clearly what happened. This satisfies CLI-05 ("User can approve or reject drafts from CLI"). The draft will show as "sent" in the dashboard. If user needs actual WhatsApp delivery, they use the dashboard. Document this limitation.

   **`drafts reject <id>`:**
   - Find the draft, verify pending
   - `db.update(drafts).set({ status: 'rejected', actionedAt: Date.now() }).where(and(eq(drafts.id, id), eq(drafts.status, 'pending'))).run()`
   - Verify the update affected a row (check `.changes` property from better-sqlite3 RunResult). If 0 changes, print: "Draft not found or not pending"
   - Print: `Draft <id> rejected`

   For both approve and reject: accept partial ID match. If the user provides a short prefix (e.g., first 8 chars of UUID), query with `LIKE` to find the full ID. This is a major UX win for CLI. Use: `db.select().from(drafts).where(sql\`${drafts.id} LIKE ${id + '%'} AND ${drafts.status} = 'pending'\`).all()`. If exactly 1 match, proceed. If 0, error "no matching draft". If >1, error "ambiguous ID, be more specific" and list matches.

   IMPORTANT: All DB access uses `db` from `../db.js`. Import schema tables from `../../src/db/schema.js`. Do NOT import query functions from `src/db/queries/drafts.ts` (they import from src/db/client.ts which imports config.ts).

1. Create `cli/commands/calendar.ts`:
   - Export `addCalendarCommand(program: Command): void`
   - Create `calendar` parent command. Under it, create `members` subcommand group with 3 sub-subcommands:

   **`calendar members list --group <jid>`:**
   - `--group <jid>` is a required option
   - Normalize JID: if it doesn't contain `@`, append `@g.us`
   - Query the group: `db.select().from(groups).where(eq(groups.id, jid)).get()`
   - If not found, print error: "Group <jid> not found" and exit 1
   - Parse `memberEmails` field: `JSON.parse(group.memberEmails || '[]')` — this is stored as a JSON array string
   - If empty, print: "No member emails configured for <group name or jid>"
   - Otherwise, print each email on its own line with index number, e.g.:
     ```
     Member emails for <group name> (<jid>):
       1. alice@example.com
       2. bob@example.com
     ```
   - Use renderToString with a simple Ink component for colored output

   **`calendar members add --group <jid> <email>`:**
   - `<email>` is a required argument, `--group <jid>` is a required option
   - Normalize JID
   - Fetch group, error if not found
   - Parse existing emails array
   - Check if email already exists (case-insensitive). If so, print: "Email already in list" and exit 0
   - Add email to array
   - Serialize back to JSON: `JSON.stringify(emails)`
   - Update: `db.update(groups).set({ memberEmails: serialized, updatedAt: Date.now() }).where(eq(groups.id, jid)).run()`
   - Print: `Added <email> to group <name or jid> (now <N> members)`

   **`calendar members remove --group <jid> <email>`:**
   - Same setup as add
   - Parse existing emails, filter out the target (case-insensitive match)
   - If email was not in list, print: "Email not found in list" and exit 0
   - Serialize and update
   - Print: `Removed <email> from group <name or jid> (now <N> members)`

   Commander structure for nested subcommands:
   ```typescript
   const calendarCmd = new Command('calendar').description('Calendar management');
   const membersCmd = new Command('members').description('Manage group member emails');

   membersCmd.command('list').requiredOption('-g, --group <jid>', 'Group JID').action(...)
   membersCmd.command('add <email>').requiredOption('-g, --group <jid>', 'Group JID').action(...)
   membersCmd.command('remove <email>').requiredOption('-g, --group <jid>', 'Group JID').action(...)

   calendarCmd.addCommand(membersCmd);
   program.addCommand(calendarCmd);
   ```
