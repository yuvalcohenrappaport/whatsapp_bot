    **Helper** (file-scope) for the already-handled error envelope — call sites need this in 3 of 4 routes:
    ```ts
    function alreadyHandledReply(
      reply: import('fastify').FastifyReply,
      row: Actionable,
    ): ReturnType<FastifyReply['send']> {
      return reply.status(409).send({
        error: 'already_handled',
        currentStatus: row.status,
        actionable: row,
      });
    }
    ```

    **Route 1 — POST /api/actionables/:id/approve**
    ```ts
    fastify.post<{ Params: { id: string } }>(
      '/api/actionables/:id/approve',
      { onRequest: [fastify.authenticate] },
      async (request, reply) => {
        const { id } = request.params;
        const row = getActionableById(id);
        if (!row) return reply.status(404).send({ error: 'Actionable not found' });
        if (row.status !== 'pending_approval') return alreadyHandledReply(reply, row);
        const sock = getState().sock;
        if (!sock) return reply.status(503).send({ error: 'bot_disconnected' });
        try {
          await approveActionable(sock, row);
        } catch (err) {
          // updateActionableStatus throws on invalid transition — means a
          // concurrent actor flipped the row between our getActionableById
          // and the approve call. Re-read + surface as already_handled.
          const fresh = getActionableById(id);
          if (fresh && fresh.status !== 'pending_approval') return alreadyHandledReply(reply, fresh);
          throw err; // unknown failure — let Fastify 500
        }
        const fresh = getActionableById(id);
        return { actionable: fresh };
      },
    );
    ```

    **Route 2 — POST /api/actionables/:id/reject**
    Mirror Route 1 but call `rejectActionable(sock, row)` instead of `approveActionable`. Same 404 / 409 / 503 / race-catch discipline.

    **Route 3 — POST /api/actionables/:id/edit**
    ```ts
    fastify.post<{ Params: { id: string }; Body: { task?: string } }>(
      '/api/actionables/:id/edit',
      { onRequest: [fastify.authenticate] },
      async (request, reply) => {
        const { id } = request.params;
        const rawTask = (request.body?.task ?? '').trim();
        if (!rawTask) return reply.status(400).send({ error: 'task is required' });
        if (rawTask.length > EDIT_TASK_MAX_LEN) {
          return reply.status(400).send({
            error: 'task too long',
            maxLength: EDIT_TASK_MAX_LEN,
          });
        }
        const row = getActionableById(id);
        if (!row) return reply.status(404).send({ error: 'Actionable not found' });
        if (row.status !== 'pending_approval') return alreadyHandledReply(reply, row);
        const sock = getState().sock;
        if (!sock) return reply.status(503).send({ error: 'bot_disconnected' });

        // Mirror approvalHandler.applyDirective lines 154-163: rewrite task
        // text FIRST, then fall through to approve so one echo is sent with
        // the edited title. Re-read so the approveActionable call sees the
        // fresh task.
        updateActionableTask(id, rawTask);
        const refreshed = getActionableById(id);
        if (!refreshed) return reply.status(404).send({ error: 'Actionable disappeared' });
        try {
          await approveActionable(sock, refreshed);
        } catch (err) {
          const finalRow = getActionableById(id);
          if (finalRow && finalRow.status !== 'pending_approval') return alreadyHandledReply(reply, finalRow);
          throw err;
        }
        return { actionable: getActionableById(id) };
      },
    );
    ```

    **Route 4 — POST /api/actionables/:id/unreject**
    ```ts
    fastify.post<{ Params: { id: string } }>(
      '/api/actionables/:id/unreject',
      { onRequest: [fastify.authenticate] },
      async (request, reply) => {
        const { id } = request.params;
        const row = getActionableById(id);
        if (!row) return reply.status(404).send({ error: 'Actionable not found' });
        if (row.status !== 'rejected') return alreadyHandledReply(reply, row);
        const sock = getState().sock;
        // sock is NOT strictly required (unrejectActionable is silent per
        // CONTEXT §Undo) but kept for signature symmetry — we still require
        // the bot to be connected so a subsequent WhatsApp approval can fire.
        if (!sock) return reply.status(503).send({ error: 'bot_disconnected' });
        try {
          await unrejectActionable(sock, row, UNREJECT_GRACE_MS);
        } catch (err) {
          if (err instanceof GraceExpiredError) {
            return reply.status(409).send({
              error: 'grace_expired',
              graceMs: UNREJECT_GRACE_MS,
              actionable: row,
            });
          }
          // Transition throw means the row got re-handled between the guard
          // and the call — surface as already_handled.
          const fresh = getActionableById(id);
          if (fresh && fresh.status !== 'rejected') return alreadyHandledReply(reply, fresh);
          throw err;
        }
        return { actionable: getActionableById(id) };
      },
    );
    ```

    **Ordering / placement**: Insert the four new routes AFTER the existing DELETE handler (around line 323) so reviewers see the write surface grouped. Do not reorder existing routes.

    **Race discipline is the whole point** — every mutation route does the triple check:
      1. `getActionableById` → 404 if missing
      2. status pre-check → 409 `already_handled` if not in the expected source status
      3. call the primitive; if it throws (updateActionableStatus's invalid-transition guard), re-read and surface as 409 if the status drifted — otherwise rethrow for Fastify to 500

    The SSE stream (from Plan 43-01 line 177) picks up the status change on the next 3s poll — no new emit plumbing. Client optimism + server SSE is the "≤3s" of SC#3 per CONTEXT.

    **Do NOT:**
    - Add a Zod schema file for request envelopes — the plugin uses inline runtime checks (same convention as lines 216-245 for PATCH/:id and 247-293 for POST / route). Adding a Zod layer would break consistency.
    - Emit an extra SSE frame manually — the hash-poll handles it (verified in Plan 43-01 test case 10).
    - Forward user-supplied `task` > 500 chars — the self-chat echo would spam the owner's own chat with a novel. Enforce `EDIT_TASK_MAX_LEN = 500`.
    - Swallow `approveActionable` / `rejectActionable` errors other than the transition-race one. Those primitives already swallow Google Tasks / Gemini failures internally per Phase 42 design; any residual error is a real server bug and should 500.
