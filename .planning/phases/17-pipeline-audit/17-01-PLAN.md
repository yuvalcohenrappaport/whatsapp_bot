<task type="auto">
  <name>Task 2: Fix URL quality -- extract from grounding metadata</name>
  <files>src/groups/travelSearch.ts</files>
  <action>
In `travelSearch.ts`, modify the `geminiGroundedSearch()` function to extract URLs from the Gemini grounding metadata instead of relying on AI-generated URLs in the text response. This is a minimal patch because Phase 20 will rewrite travelSearch.ts entirely.

**Current behavior:** The function reads `response.text`, parses it as JSON, and uses the `url` field from the parsed objects. These URLs are AI-generated and may be hallucinated.

**New behavior:** After getting the response, check `response.candidates?.[0]?.groundingMetadata?.groundingChunks` for real URLs. Use these to replace or supplement the text-parsed URLs.

**Specific changes to `geminiGroundedSearch()`:**

1. After the existing `const response = await ai.models.generateContent(...)` call, extract grounding URLs:
   - Access `response.candidates?.[0]?.groundingMetadata?.groundingChunks`
   - Filter to chunks that have a `web` property with a non-empty `uri`
   - Build an array of `{ uri: string, title: string }` from `chunk.web.uri` and `chunk.web.title`

2. After the existing JSON parsing logic (which produces the `results` array), cross-reference grounding URLs with parsed results:
   - For each parsed result, check if any grounding chunk's title loosely matches the result title (case-insensitive substring match). If so, replace the result's `url` with the grounding chunk's `uri`.
   - If no title match is found, but there are unused grounding URLs remaining, assign them to results that have empty or suspiciously short URLs (less than 20 chars) in order.
   - If grounding metadata is not available (null/undefined/empty), keep the text-parsed URLs as-is (this is the fallback path).

3. Log at debug level: how many grounding chunks were found, how many were matched to results, how many results still use text-parsed URLs.

**Why this approach:** The text-parsed results have better structure (title, snippet, price) because Gemini formats them per our prompt. The grounding metadata has better URLs because they come from actual Google Search results. We combine both: structure from text, URLs from metadata. This is a minimal patch that improves URL quality without changing the function signature or return type.

**Do NOT change:**
- The function signature or return type (`SearchResult[]`)
- The `knowledgeFallback()` function (it has no grounding metadata by design)
- The `searchTravel()` function (it just calls both and picks)
- Any other file
  </action>

2. Run `npx tsc --noEmit` to verify no TypeScript errors were introduced.
  </verify>
  <done>
The `geminiGroundedSearch()` function extracts URLs from `response.candidates[0].groundingMetadata.groundingChunks` when available, falling back to text-parsed URLs when grounding metadata is absent. URL quality is improved without changing the module's public API.

<task type="auto">
  <name>Task 3: Fix follow-up context framing for reply chain</name>
  <files>src/groups/travelHandler.ts</files>
  <action>
In `travelHandler.ts`, modify the follow-up context injection (around lines 178-186) to explicitly tell the intent parser that this is a follow-up search. This prevents Gemini from parsing terse follow-ups like "show cheaper" or "יותר זול" as non-travel or vague.

**Current behavior:** When `priorContext` exists, the handler prepends the previous query and results to `recentContext` but does not tell the parser "this is a follow-up."

**New behavior:** When `priorContext` exists, add framing text that tells Gemini this is a follow-up refinement.

**Specific change:**

Find the block that starts with `if (priorContext) {` and modifies `recentContext`. Change the prepended text to include explicit follow-up framing. The recentContext assignment should become:

```
recentContext =
  `[FOLLOW-UP SEARCH] This message is a reply to a previous travel search result. ` +
  `The user is refining or continuing their search. Treat this as travel-related.\n` +
  `Previous search query: ${priorContext.query}\n` +
  `Previous results:\n${priorContext.results}\n\n` +
  `User follow-up message: ${msg.body}\n\n` +
  recentContext;
```

Also, when `priorContext` exists, change the call to `parseTravelIntent` so the `messageText` argument is augmented. Currently it passes `msg.body` which might be just "show cheaper". Instead, when priorContext exists, pass a combined string: `"Follow-up to: ${priorContext.query}. User says: ${msg.body}"` as the first argument. This gives the parser both the original context and the follow-up in the primary message field, not just in the recentContext.

**Do NOT change:**
- The non-follow-up path (when priorContext is null)
- The reply chain detection logic (quotedMessageId lookup in travelResultMessages)
- The rate limiting or "Searching..." indicator logic
- Any other file

**Why this is minimal:** This changes only the text content passed to parseTravelIntent, not the function signatures, not the control flow. Phase 18 will modify travelHandler for history_search dispatch, which is a different code path.

  </verify>
