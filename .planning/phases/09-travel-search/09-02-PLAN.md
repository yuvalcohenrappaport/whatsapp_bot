**1. Create `src/groups/travelSearch.ts` — Google scraping + Gemini knowledge fallback:**

Export a `SearchResult` interface:
```typescript
export interface SearchResult {
  title: string;
  url: string;
  snippet: string;
  price: string | null;
}
```

**Primary: `scrapeGoogleResults(query: string): Promise<SearchResult[]>`**

Use native `fetch` (Node 20 built-in) with AdsBot-Google user-agent to get static Google SERP HTML:
- URL: `https://www.google.com/search?q=${encodeURIComponent(query)}&hl=en`
- Headers: `User-Agent: 'AdsBot-Google (+http://www.google.com/adsbot.html)'`, `Accept: 'text/html,...'`, `Accept-Language: 'en-US,en;q=0.9'`
- Parse with `import { load } from 'cheerio'`

Multi-selector cascade (Google changes class names frequently — try multiple):
1. Primary: `.yuRUbf > a` — extract href, find `h3` text for title, find `.VwiC3b` or `.IsZvec` in closest `.g` ancestor for snippet
2. Fallback: broad `h3` scan — find closest `a` ancestor, extract href (only if starts with `http`), use h3 text as title
3. If both yield 0 results, return empty array

For each result, attempt price extraction from snippet: look for currency patterns (`$`, `€`, `₪`, `USD`, `ILS` followed by digits, or digits followed by currency symbols). Store matched price or null.

Limit to 3 results (per locked decision).

Log which selector tier produced results (pino logger). Log warning if response HTML < 5000 chars (likely blocked/redirect per pitfall 1).

**Fallback: `geminiKnowledgeFallback(searchQuery: string, lang: 'he' | 'en'): Promise<SearchResult[]>`**

When scraping returns 0 results or throws, ask Gemini (without grounding tools) to provide 3 travel recommendations:
- System instruction: "You are a travel assistant. Provide exactly 3 recommendations for the given query. For each, give a name, brief description, and estimated price range if you know it. Note: these are general knowledge recommendations, not live prices. Respond as a JSON array of objects with fields: title, snippet, price (string or null). Respond in {lang}."
- Parse response JSON. Return as SearchResult[] with `url: ''` (no live links from knowledge fallback).

**Main export: `searchTravel(searchQuery: string, lang: 'he' | 'en'): Promise<{ results: SearchResult[]; isFallback: boolean }>`**
- Try `scrapeGoogleResults(searchQuery)` first
- If results.length > 0, return `{ results, isFallback: false }`
- If results.length === 0 or scraping throws, log the failure, call `geminiKnowledgeFallback(searchQuery, lang)`, return `{ results, isFallback: true }`

Export `formatTravelResults(results: SearchResult[], lang: 'he' | 'en', isFallback: boolean): string`

Format as rich card-style message per locked decision:

  {i}. *{title}*

- If result has no URL (fallback), omit the URL line
- Keep cards concise — 3 results keeps group chat clean

Export `formatHelpText(botDisplayName: string, lang: 'he' | 'en'): string` — the casual help text for non-travel mentions. Include 3 example lines using the actual bot display name. Keep tone casual and friendly per locked decision.

**1. Replace the Plan 01 placeholder with actual search + format pipeline in `travelHandler.ts`:**

Import `searchTravel` from `./travelSearch.js` and `formatTravelResults`, `formatHelpText` from `./travelFormatter.js`.

In the `handleTravelMention` function, replace the placeholder "Travel search coming soon!" block with:
- Call `searchTravel(intent.searchQuery!, lang)` (searchQuery is non-null when isTravelRelated=true and isVague=false)
- Call `formatTravelResults(results, lang, isFallback)` to build the message
- Send the formatted message to the group via `sock.sendMessage(groupJid, { text: formattedMessage })`
- Capture the sent message key: `const sent = await sock.sendMessage(...)`
- Store in `travelResultMessages` Map: `sent.key.id -> { query: intent.searchQuery, results: formattedMessage, groupJid }`

Also update the help text to use `formatHelpText(botDisplayName, lang)` instead of inline text.

**2. Wire reply chain context (follow-up requests):**

At the top of `handleTravelMention`, before the `isBotMentioned` check, add reply chain detection:
- If `quotedMessageId` is not null AND `travelResultMessages.has(quotedMessageId)`:
  - This is a follow-up to a previous travel search — treat it as a travel mention regardless of whether bot is @mentioned
  - Retrieve the original `{ query, results }` from the Map
  - Append them to the recent context string passed to `parseTravelIntent`: `"Previous search query: {query}\nPrevious results:\n{results}\n\nUser follow-up: {msg.body}"`
  - This ensures Gemini understands the follow-up context (per locked decision)
  - Skip the `isBotMentioned` check and proceed directly to "Searching..." + parse + search

**3. Add per-group rate limiting:**

Create a module-level `lastRequestTime` Map<string, number> (groupJid -> timestamp ms).

Before sending "Searching...", check if the group has made a travel request in the last 30 seconds:
- If `Date.now() - lastRequestTime.get(groupJid) < 30_000`:
  - Send a brief rate limit message: "Please wait a moment before the next search." / "בבקשה המתינו רגע לפני החיפוש הבא."
  - Return true (handled, but throttled)
- Otherwise, update `lastRequestTime.set(groupJid, Date.now())` and proceed

**4. Limit travelResultMessages Map size:**

To prevent unbounded memory growth, cap the Map at 500 entries. When inserting a new entry and size >= 500, delete the oldest entry (first key via `Map.keys().next().value`). Travel result message IDs are ephemeral — losing old ones just means those replies won't be recognized as follow-ups.

**5. Wrap the entire search+format+send flow in try/catch:**

If search or formatting fails, send an error message to the group: "Sorry, something went wrong with the search. Please try again." / "סליחה, משהו השתבש בחיפוש. נסו שוב." — never crash the pipeline.
