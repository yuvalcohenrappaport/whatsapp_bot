<task type="auto">
  <name>Task 1: Create /api/actionables REST + SSE plugin</name>
  <files>src/api/routes/actionables.ts</files>
  <action>
    Create a new Fastify plugin exporting a default async function `actionablesRoutes(fastify: FastifyInstance)`. Register three routes, all JWT-gated in the exact style of `src/api/routes/reminders.ts` (for REST) and `src/api/linkedin/routes/stream.ts` (for SSE).

    **Route 1 — GET /api/actionables/pending**
    - Guard: `{ onRequest: [fastify.authenticate] }`
    - Body: `{ actionables: Actionable[] }` — result of `getPendingActionables()` (already ordered by `detectedAt` desc by the query layer).
    - Pass the row objects through verbatim (no field stripping). The dashboard's Zod schema will validate shape on the client.

    **Route 2 — GET /api/actionables/recent**
    - Guard: `{ onRequest: [fastify.authenticate] }`
    - Querystring: `limit` (optional, default 50, clamp to 1..200).
    - Body: `{ actionables: Actionable[] }` — result of `getRecentTerminalActionables(limit)` (already exists in `src/db/queries/actionables.ts` line 216, ordered by `updatedAt` desc, filters to `['approved','rejected','expired','fired']`).
    - Parse limit with `parseInt(raw, 10)`; on NaN default to 50; clamp to [1, 200].

    **Route 3 — GET /api/actionables/stream** (SSE)
    - JWT via `?token=<jwt>` query string. EventSource cannot send headers. Copy the exact auth block from `src/api/linkedin/routes/stream.ts` lines 75-81: read `request.query.token`, call `fastify.jwt.verify(token ?? '')`, on throw return `reply.status(401).send({ error: 'Unauthorized' })`.
    - SSE framing headers + `flushHeaders()`: `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`, `X-Accel-Buffering: no` (all four — matches LinkedIn stream.ts).
    - Poll loop every 3 seconds (`POLL_INTERVAL_MS = 3_000`). Inside the loop, read both lists (`getPendingActionables()` + `getRecentTerminalActionables(50)`), compute a combined stable sha1 hash over `[id, status, updatedAt, enrichedTitle?.slice(0,50) ?? null, todoTaskId]` per row across both lists, concat-then-hash. Emit `event: actionables.updated\ndata: {JSON}\n\n` with body `{ pending: [...], recent: [...] }` on hash change. FIRST poll ALWAYS emits (seeds the client).
    - Heartbeat `: ping\n\n` every 15 seconds regardless of content changes.
    - On `request.raw.on('close')`: clear both intervals and mark `closed = true`.
    - On DB read error: log `fastify.log.warn({err}, '[actionables-stream] poll failed; will retry')`; do NOT crash; do NOT emit an event; next tick may succeed.
    - Use `createHash('sha1')` from `node:crypto`. Export the hash helper as `hashActionables(pending, recent): string` so the vitest suite can assert stability.

    Imports (example):
    ```ts
    import type { FastifyInstance } from 'fastify';
    import { createHash } from 'node:crypto';
    import {
      getPendingActionables,
      getRecentTerminalActionables,
      type Actionable,
    } from '../../db/queries/actionables.js';
    ```

    Constants inside the module:
    - `const POLL_INTERVAL_MS = 3_000;`
    - `const HEARTBEAT_INTERVAL_MS = 15_000;`
    - `const DEFAULT_RECENT_LIMIT = 50;`
    - `const MAX_RECENT_LIMIT = 200;`

    Do NOT add a second event name. ONE event type: `actionables.updated`. Payload shape is fixed at `{ pending: Actionable[], recent: Actionable[] }`.

    Do NOT thread emit calls into approvalHandler / detectionService / expiryScan — the hash-polling loop catches every status change automatically (same tradeoff the LinkedIn stream made per its docstring "Localhost calls are ~5ms; 3s polling = 0.17% CPU").
  </action>
