**New function: `sendPreSendNotification(id: string)`**
1. Read the message via `getScheduledMessageById(id)`. Return early if not found or status !== 'pending'.
2. Get recipients via `getRecipientsForMessage(id)`.
3. Build recipient display: single recipient uses `getContact(jid)?.name ?? jid`. Multiple (<=3) list comma-separated names. More than 3 uses "N recipients".
4. Build content preview: split content on whitespace, take first 20 words, append "..." if truncated.
5. Build type label: 'text' -> 'Text', 'voice' -> 'Voice', else 'AI'.
6. Format send time using `new Intl.DateTimeFormat('en-IL', { hour: '2-digit', minute: '2-digit', timeZone: 'Asia/Jerusalem' }).format(new Date(msg.scheduledAt))`.
7. Build notification text (emoji-labeled card):
   ```
   Scheduled message in 10 min
   [recipient]
   [type]
   "[preview]"
   [time]
   Reply to this message with "cancel" to stop it.
   ```
   Use the emoji labels from CONTEXT: calendar, person, memo, speech, clock emojis.
8. Get sock from `getState().sock`. If no sock: log warning, fall back to scheduling fire directly at `msg.scheduledAt` via `scheduleMessage(id, msg.scheduledAt, cb)` — no cancel window but message still sends.
9. Try sending notification via `await sock.sendMessage(config.USER_JID, { text: notification })`. Do NOT use `sendWithTimeout` — self-chat notifications use plain sendMessage (best-effort).
10. Capture `sent?.key?.id` and call `updateScheduledMessageNotificationMsgId(id, sentMsgId)`.
11. Set status to 'notified': `updateScheduledMessageStatus(id, 'notified')`.
12. Schedule the actual fire: `const actualFireAt = Math.max(msg.scheduledAt, Date.now() + NOTIFICATION_LEAD_MS);` then `scheduleMessage(id, actualFireAt, fireCallback)`.

Actually, re-reading the research more carefully: the notification should fire at `scheduledAt - 10min`, not at `scheduledAt`. The scheduler's `scheduleMessage(id, timestamp, cb)` sets a timer for `timestamp`. So the correct approach for `scheduleAllUpcoming` and periodic scan is to schedule the notification timer at `scheduledAt - NOTIFICATION_LEAD_MS`:

In `initScheduledMessageScheduler`:
- `startPeriodicScan` callback: receives an id. This callback fires when the timer expires. The scheduler internally reads `msg.scheduledAt` to set the timer. Since we want notification at `scheduledAt - 10min`, we need to modify how we schedule. But `startPeriodicScan` passes messages to `scheduleMessage(id, msg.scheduledAt, cb)` internally. We CANNOT change that from outside.

RESOLUTION: Keep the scheduler firing at `msg.scheduledAt` via the existing `scheduleMessage` call. The `onFire` callback becomes a dispatcher: if message is 'pending', run `sendPreSendNotification` which sends notification and schedules the actual send 10 minutes later. If message is 'notified', run `fireMessage`. This means for messages picked up by the periodic scan, the send is delayed by 10 minutes past the original scheduledAt — which is exactly what CONTEXT.md requires ("still follows the full pipeline").

So in `initScheduledMessageScheduler`:
```typescript
const dispatchCallback = (id: string) => {
  const msg = getScheduledMessageById(id);
  if (!msg) return;
  if (msg.status === 'pending') {
    sendPreSendNotification(id).catch(err =>
      logger.error({ err, id }, 'Unhandled error in notification dispatch'));
  } else if (msg.status === 'notified') {
    fireMessage(id).catch(err =>
      logger.error({ err, id }, 'Unhandled error in fire dispatch'));
  }
};
```
Use `dispatchCallback` for `startPeriodicScan`, `scheduleAllUpcoming`, and recovery stagger.
