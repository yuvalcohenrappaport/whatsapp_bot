## Discretion Decisions

**Per-source parallel initial-load (CONTEXT §5):** Rather than render a single full-page skeleton until all three sources resolve, the hook fires three parallel fetches to the per-source routes Plan 44-03 exposes and flips each slice to `'ok'` independently. Each source's pills appear the moment its own fetch resolves. SSE layers on top only after at least one source is live — the unified envelope then replaces the per-source slices. This is Option A from the phase-44 planning pass: smallest diff, cheapest client code, matches the "each source loads independently" lock in CONTEXT §5.

Alternative rejected: server-side SSE per-source (Option B). Would require splitting `/api/calendar/stream` into three channels or adding per-source discriminators to the frame — more reconnect churn, more Zod schema duplication, higher complexity for zero user-visible win (live updates are low-volume).

    Declare the discriminated-union schema matching Plan 44-03's server envelope exactly.

    ```ts
    import { z } from 'zod';

    const BaseItem = {
      id: z.string(),
      title: z.string(),
      start: z.number(),
      end: z.number().nullable(),
      isAllDay: z.boolean(),
      language: z.enum(['he', 'en', 'mixed']),
      sourceFields: z.record(z.string(), z.unknown()),
    } as const;

    export const CalendarItemSchema = z.discriminatedUnion('source', [
      z.object({ source: z.literal('task'), ...BaseItem }),
      z.object({ source: z.literal('event'), ...BaseItem }),
      z.object({ source: z.literal('linkedin'), ...BaseItem }),
    ]);
    export type CalendarItem = z.infer<typeof CalendarItemSchema>;

    export const SourceStatusSchema = z.enum(['ok', 'error']);

    export const CalendarEnvelopeSchema = z.object({
      items: z.array(CalendarItemSchema),
      sources: z.object({
        tasks: SourceStatusSchema,
        events: SourceStatusSchema,
        linkedin: SourceStatusSchema,
      }),
    });
    export type CalendarEnvelope = z.infer<typeof CalendarEnvelopeSchema>;
    ```

    **File B — `dashboard/src/hooks/useCalendarStream.ts`:**

    Topology: per-source parallel initial-load + unified SSE on top. CONTEXT §5 requires that each source render independently — no source waits for the slowest. So the hook owns both phases.

    **State shape:**
    ```ts
    type SourceSlice<T> = { items: T[]; status: 'loading' | 'ok' | 'error' };
    type UseCalendarStreamResult = {
      tasks: SourceSlice<CalendarItem>;
      events: SourceSlice<CalendarItem>;
      linkedin: SourceSlice<CalendarItem>;
      sseStatus: 'idle' | 'connecting' | 'open' | 'reconnecting' | 'error';
    };
    ```

    **Phase 1 — parallel initial load (on mount):**
    Fire THREE independent `fetch` calls via `apiFetch` (dashboard's existing JWT wrapper), each against a Plan 44-03 per-source route. Each response shape is `{items: CalendarItem[]}` which is safeParsed and flipped to `status: 'ok'` on success or `status: 'error'` on parse/network failure. A rejected promise on one source does NOT affect the other two. Match the existing dashboard convention — bare `useEffect + fetch` per-source (the PendingTasks page uses this pattern, not React Query), one effect per source for clarity.

    ```ts
    useEffect(() => {
      let cancelled = false;
      apiFetch('/api/actionables/with-due-dates').then(r => r.json()).then(json => {
        if (cancelled) return;
        const parsed = z.object({ items: z.array(CalendarItemSchema) }).safeParse(json);
        setTasks(parsed.success ? { items: parsed.data.items, status: 'ok' } : { items: [], status: 'error' });
      }).catch(() => { if (!cancelled) setTasks({ items: [], status: 'error' }); });
      return () => { cancelled = true; };
    }, []);
    // Same effect shape twice more for events + linkedin.
    ```

    **Phase 2 — unified SSE (opened after any source resolves):**
    Model 1:1 on `useActionablesStream.ts` — new EventSource against `/api/calendar/stream`, listen for `calendar.updated`, safeParse with `CalendarEnvelopeSchema`, then SPLIT the envelope back into the three source slices (filter items by `.source`) and set all three slices in one batch. From this point on, per-source loading/error states are frozen at their post-fetch values; the SSE envelope drives all subsequent updates. If the SSE connection drops, `sseStatus` flips to `reconnecting` — but per-source slices keep their last-known items (no re-skeleton).

    Polling fallback (10s, for schema-drift safety): polls the unified `/api/calendar/items` (not the per-source routes — once past initial load, a unified fetch is cheaper than three). Mirrors the actionables hook's fallback pattern.

    **Why this shape (vs React Query):** PendingTasks.tsx uses bare `useEffect + fetch`, not React Query, so this hook matches codebase convention. If future tickets migrate that surface to React Query, this hook trivially becomes `const tasksQ = useQuery(...); const eventsQ = useQuery(...); const linkedinQ = useQuery(...)` — refactor cost is low.

    **File C — `dashboard/src/lib/ist.ts`:**

    Pure-function IST helpers. All locked to `Asia/Jerusalem` via `toLocaleString('en-GB', {timeZone: 'Asia/Jerusalem'})` + a regex-based reformat to `YYYY-MM-DD HH:mm`. Exports:

    - `formatIstAbsolute(ms: number): string` → `"2026-04-20 14:32"`.
    - `formatIstTime(ms: number): string` → `"14:32"`.
    - `formatIstDateShort(ms: number): string` → `"Tue 20 Apr"`.
    - `startOfIstWeek(ms: number): number` → Sunday 00:00 IST (returns unix ms). Use: parse ms as IST date, subtract (localDow % 7) days, set time to 00:00.
    - `addIstDays(ms: number, n: number): number` → adds n × 86400000 ms (works because IST has no DST — Israel is UTC+2/UTC+3 but for consistent grid math we accept a day's worth of ms; DST transitions are handled by the user tolerating a 1-hour visual shift twice a year, same tradeoff FullCalendar makes).
    - `istTodayAtMs(hour: number, minute: number): number` → returns today's 00:00 IST offset + hour:minute.
    - `istDayStartMs(ms: number): number` → midnight of the IST day that `ms` falls in.
    - `sameIstDay(aMs: number, bMs: number): boolean` → true if both fall in the same IST calendar day.
