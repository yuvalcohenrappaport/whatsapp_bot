Write the backfill migration with explicit INSERT...SELECT for each legacy-source × legacy-status combination per 39-CONTEXT.md. Each INSERT is guarded by a `WHERE NOT EXISTS` clause so the migration is safe to re-run.

Structure:

```sql
-- Phase 39 backfill — migrate pending & terminal rows from reminders + todoTasks
-- into the unified actionables table. Idempotent via NOT EXISTS guards.

-- ─── reminders(source='commitment') ─────────────────────────────

-- pending commitments → approved (already pushed to Google Tasks)
INSERT INTO `actionables` (
  id, source_type, source_contact_jid, source_contact_name,
  source_message_id, source_message_text, detected_language,
  original_detected_task, task, status,
  detected_at, fire_at,
  enriched_title, enriched_note,
  todo_task_id, todo_list_id,
  approval_preview_message_id,
  created_at, updated_at
)
SELECT
  r.id,
  'commitment',
  COALESCE(r.source_contact_jid, 'unknown'),
  NULL, -- contact name was not snapshotted in legacy reminders
  NULL, NULL, 'en', -- no source message id/text/lang snapshot in legacy rows
  r.task, r.task,
  'approved', -- grandfather: already in Google Tasks
  r.created_at, r.fire_at,
  NULL, NULL,
  r.todo_task_id, r.todo_list_id,
  NULL,
  r.created_at, r.updated_at
FROM `reminders` r
WHERE r.source = 'commitment' AND r.status = 'pending'
  AND NOT EXISTS (SELECT 1 FROM `actionables` a WHERE a.id = r.id);
--> statement-breakpoint

-- fired commitments → fired (terminal)
INSERT INTO `actionables` (...)
SELECT ..., 'fired', ... FROM `reminders` r
WHERE r.source = 'commitment' AND r.status = 'fired'
  AND NOT EXISTS (SELECT 1 FROM `actionables` a WHERE a.id = r.id);
--> statement-breakpoint

-- cancelled commitments → rejected
INSERT INTO `actionables` (...)
SELECT ..., 'rejected', ... FROM `reminders` r
WHERE r.source = 'commitment' AND r.status = 'cancelled'
  AND NOT EXISTS ...;
--> statement-breakpoint

-- skipped commitments → expired
INSERT INTO `actionables` (...)
SELECT ..., 'expired', ... FROM `reminders` r
WHERE r.source = 'commitment' AND r.status = 'skipped'
  AND NOT EXISTS ...;
--> statement-breakpoint

-- ─── reminders(source='user' or NULL) = self-chat commands ──────

-- pending user reminders → approved (user-initiated, already acted on)
INSERT INTO `actionables` (...)
SELECT
  r.id, 'user_command',
  (SELECT value FROM settings WHERE key = 'user_jid'), -- or a literal placeholder; see Task 3 note
  'Self',
  NULL, '', 'en',
  r.task, r.task,
  'approved',
  r.created_at, r.fire_at,
  NULL, NULL,
  r.todo_task_id, r.todo_list_id,
  NULL,
  r.created_at, r.updated_at
FROM `reminders` r
WHERE (r.source = 'user' OR r.source IS NULL) AND r.status = 'pending'
  AND NOT EXISTS ...;
--> statement-breakpoint

-- fired / cancelled / skipped user reminders → terminal mapping (same shape as commitments)

-- ─── todoTasks ───────────────────────────────────────────────────

-- synced todos → approved (already in Google Tasks)
INSERT INTO `actionables` (...)
SELECT
  t.id, 'task',
  t.contact_jid, t.contact_name,
  NULL, COALESCE(t.original_text, ''), 'en',
  t.task, t.task,
  'approved',
  t.created_at, NULL,
  NULL, NULL,
  t.todo_task_id, t.todo_list_id,
  t.notification_msg_id, -- reuse as approval preview id for audit trail
  t.created_at, COALESCE(t.synced_at, t.created_at)
FROM `todo_tasks` t
WHERE t.status = 'synced'
  AND NOT EXISTS ...;
--> statement-breakpoint

-- pending todos → pending_approval (re-gate: user will see them in v1.8's preview flow)
INSERT INTO `actionables` (...)
SELECT ..., 'pending_approval', ... FROM `todo_tasks` t
WHERE t.status = 'pending' AND NOT EXISTS ...;
--> statement-breakpoint

-- failed todos → pending_approval (same re-gate)
INSERT INTO `actionables` (...)
SELECT ..., 'pending_approval', ... FROM `todo_tasks` t
WHERE t.status = 'failed' AND NOT EXISTS ...;
--> statement-breakpoint

-- cancelled todos → rejected (terminal)
INSERT INTO `actionables` (...)
SELECT ..., 'rejected', ... FROM `todo_tasks` t
WHERE t.status = 'cancelled' AND NOT EXISTS ...;
```

**USER_JID resolution note:**
The `user_command` backfill needs `config.USER_JID` as `source_contact_jid`. SQL can't read env vars / config files. Two options:
1. Read the value from `settings` table if it's stored there (check `src/db/queries/settings.ts`)
2. Use a placeholder string like `'USER_JID_PLACEHOLDER'` and let a one-shot startup fixup in `index.ts` rewrite it to the real `config.USER_JID` on first boot (idempotent: `UPDATE actionables SET source_contact_jid = ? WHERE source_contact_jid = 'USER_JID_PLACEHOLDER'`)

Pick option 2 for deterministic SQL — add the fixup in a small follow-up TS snippet inside the existing reminder-system init (document this in the plan summary). Option 1 is cleaner if `user_jid` is already in settings; verify and use it if present.
