## Milestone Goal

Extend the v1.4 Travel Agent from reactive-search + basic-memory into a trip-aware concierge that ingests multimedia, remembers per-person context, detects conflicts, pushes day-of intelligence, and exposes a trip dashboard — validated against a real Italy trip 2+ months out.

### Phase 53 — Smarter Search (Restaurants)

- [ ] **SRCH2-01**: `travelParser` has a `queryType='restaurants'` branch triggered by Hebrew and English restaurant keywords. `travelSearch.geminiMapsSearch` requests restaurant-specific fields (photo_url, open_now, price_level, cuisine, reservation_url). `travelFormatter` produces compact one-liner results: `🍽️ {name} · {cuisine} · {price_tier} · {open_now} · {rating}⭐ · {url}`. Transit, attractions, and shopping search are explicitly out of scope for v2.1

### Phase 54 — Proactive Day-Of Intelligence

- [ ] **DAY-01**: A 15-min cron posts at most one briefing per active-trip per day, at `briefing_time` destination-tz (default 08:00), within the window `[start_date − 1day, end_date]`. Archived trips are skipped. `metadata.last_briefing_date` prevents duplicate posts
- [ ] **DAY-02**: Briefings enrich the calendar event list with: weather from OpenWeather (destination coords cached in `metadata.coords`); transit alerts from Gemini grounded search (`googleSearch` tool); unresolved open questions from DB; today's conflicts from `trip_decisions.conflicts_with`; budget burn from `getBudgetRollup`. Final composition by Gemini with structured input + Hebrew output instruction
- [ ] **DAY-03**: On any enrichment source failure (OpenWeather timeout, Gemini error, etc.), the briefing falls back to a minimal calendar-only template posted in Hebrew. No day is skipped because of enrichment failure

### Phase 55 — Trip Dashboard View

- [ ] **DASH-TRIP-01**: Dashboard route `/trips/:groupJid` renders trip header (destination, dates, countdown, budget target vs actual per category), vertical timeline of confirmed calendar events, Leaflet map with markers per decision that has lat/lng, decisions board grouped by category with filter-by-origin control, open-questions list, and conflict alerts. Archived trips render in read-only mode
- [ ] **DASH-TRIP-02**: Minimal-edit controls wired through JWT-gated API routes: `DELETE /api/trips/:groupJid/decisions/:id` (soft-delete via `status='deleted'`), `PATCH /api/trips/:groupJid/questions/:id/resolve`, `PATCH /api/trips/:groupJid/budget`. SSE broadcasts propagate changes to every open dashboard session within ~3s
- [ ] **DASH-TRIP-03**: Dashboard "Export" button calls `POST /api/trips/:groupJid/export` which creates a Google Doc titled `{destination} {start_date} — Trip Summary` containing trip header, timeline, decisions grouped by category, open questions, and budget summary. Uses existing Google OAuth with a new `https://www.googleapis.com/auth/documents` scope (one-time re-auth by owner during Phase 55 UAT). Returns owner-private shareable URL

## Out of Scope (explicitly deferred)

- v1.4 audit tech debt (in-memory rate-limits, 30-char prefix matching, 5-result search cap)
- Multi-trip per group (schema retains one-active-trip-per-group invariant)
- Transit / attractions / shopping search expansion
- PDF / HTML / public-link trip export variants
- Automated conflict resolution (bot only flags, doesn't rebook)
- Voice-reply for briefings
- Real-time flight / hotel price tracking
- Manual archive dashboard button (auto-archive is the only archival path)
