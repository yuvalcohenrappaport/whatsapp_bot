Before the human verification, ensure the test environment is ready:

   ```bash
   sqlite3 data/bot.db "UPDATE settings SET value = 'true' WHERE key = 'voice_replies_enabled';"
   ```
   If the row doesn't exist yet, insert it:
   ```bash
   sqlite3 data/bot.db "INSERT OR REPLACE INTO settings (key, value) VALUES ('voice_replies_enabled', 'true');"
   ```

2. Find a whitelisted contact that is in `auto` mode and enable voice for them:
   ```bash
   sqlite3 data/bot.db "SELECT jid, name, mode, voice_reply_enabled FROM contacts WHERE mode = 'auto' LIMIT 5;"
   ```
   Then enable voice for that contact:
   ```bash
   sqlite3 data/bot.db "UPDATE contacts SET voice_reply_enabled = 1 WHERE jid = '<THE_JID>';"
   ```

3. Restart the bot to pick up code changes:
   ```bash
   pm2 restart whatsapp-bot && pm2 logs whatsapp-bot --lines 20
   ```
   Verify clean startup — no import errors, no TypeScript errors.

4. Tail the logs for the voice test:
   ```bash
   pm2 logs whatsapp-bot --lines 0
   ```
   Leave this ready for the human verification step.

  <what-built>Complete voice pipeline: voice message receipt, audio download, transcription via ElevenLabs Scribe v2, AI reply via Gemini, TTS via ElevenLabs eleven_v3, PTT voice note send back to contact. Text fallback for contacts with voice disabled.</what-built>
  <how-to-verify>
**Test 1: Voice reply (happy path)**
1. From a phone (not the bot's number), open WhatsApp chat with the bot's number
2. The contact must be in `auto` mode with `voiceReplyEnabled = true` (set up in Task 1)
3. Record and send a voice message in Hebrew (e.g., say "hey ma nishma, ma kore hayom?")
4. Watch for:
   - "Recording..." indicator appears in the chat (microphone icon) — this is the `recording` presence
   - After a few seconds, a voice note bubble arrives (NOT a file/document attachment)
   - The voice note plays Hebrew audio that sounds like a natural reply to your message
5. Check bot logs (`pm2 logs whatsapp-bot`):
   - "voice message downloaded" with byte count
   - "transcription complete" with char count
   - "TTS OGG ready" with byte count
   - No errors

**Test 2: Text fallback (voice disabled)**
1. Disable voice for the test contact:
   ```bash
   sqlite3 data/bot.db "UPDATE contacts SET voice_reply_enabled = 0 WHERE jid = '<THE_JID>';"
   ```
2. Send another voice message from the same phone
3. Watch for:
   - A TEXT reply arrives (not a voice note) — the bot transcribed the voice message and replied with text
4. Check bot logs — should show download + transcription but NO "TTS OGG ready" line

**Test 3: Text path regression**
1. Send a normal text message from the same contact
2. Verify a text reply comes back as usual — no change in behavior

If all three tests pass, type "approved". If any test fails, describe the issue.
