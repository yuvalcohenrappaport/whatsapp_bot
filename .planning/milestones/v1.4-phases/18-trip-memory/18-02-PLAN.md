  <name>Task 1: Create tripContextManager.ts with pre-filter, debounce, and classifier</name>
  <files>src/groups/tripContextManager.ts</files>
  <action>
    Create `src/groups/tripContextManager.ts` as a new module. This file contains the pre-filter, its own debounce buffer (completely separate from the calendar debounce in groupMessagePipeline.ts), and the Gemini classifier that extracts trip decisions and open questions.

    Imports: `crypto` from `node:crypto`, `pino`, `config`, `generateJson` from `'../ai/provider.js'`, and from `'../db/queries/tripMemory.js'`: `getTripContext`, `upsertTripContext`, `insertTripDecision`, `getDecisionsByGroup`.

    Use Zod v4 ONLY: `import { z } from 'zod'` (NOT 'zod/v3'). Use `z.toJSONSchema()` (NOT zod-to-json-schema). This is a locked decision from STATE.md.

    **GroupMsg interface:** Same shape as the one in groupMessagePipeline.ts: `{ id: string; senderJid: string; senderName: string | null; body: string; timestamp: number }`. Do not import it (it is not exported); redeclare locally.

    **Section 1 -- Pre-filter (`hasTravelSignal`):**
    Export a pure function `hasTravelSignal(text: string): boolean`. Logic:
    - Return false if text.length < 15 (MIN_CLASSIFY_LENGTH constant)
    - Return false if text matches SKIP_PATTERNS: emoji-only up to 10 chars, or common ack words (ok, lol, haha, yes, no, and Hebrew equivalents)
    - Return true if text matches TRAVEL_SIGNALS regex: English travel keywords (hotel, hostel, airbnb, flight, fly, airport, book, reserve, booked, decided, destination, budget, itinerary, trip, travel, vacation, rent, car rental) as word boundaries, plus Hebrew keywords without word boundaries (since Hebrew has no spaces between prefix letters): the research lists these exact Hebrew roots.
    - Return false otherwise (no travel signal detected)
    The regex patterns should be module-level constants, compiled once.

    **Section 2 -- Debounce buffer:**
    Private module-level Map: `tripDebounceBuffers = new Map<string, { messages: GroupMsg[]; timer: NodeJS.Timeout }>()`. Constant: `TRIP_DEBOUNCE_MS = 10_000`.

    Export `addToTripContextDebounce(groupJid: string, msg: GroupMsg): void`. Logic:
    - Call `hasTravelSignal(msg.body)` first. If false, log at debug level and return immediately. The message never enters the buffer.
    - If true, add to the debounce buffer for this groupJid. If buffer exists, clearTimeout, push message, reset timer. If not, create new buffer with this message and set timer. Timer callback: delete buffer from map, call `processTripContext(groupJid, messages)` with `.catch()` for error logging.
    - This is the exact same debounce pattern used by `addToDebounce` in groupMessagePipeline.ts.

    **Section 3 -- Zod schema for classifier output:**
    Define `TripClassifierSchema` using Zod v4:
    - `decisions`: array of objects with `type` (enum: destination, accommodation, activity, transport, dates, budget), `value` (string), `confidence` (enum: high, medium, low)
    - `openItems`: array of objects with `question` (string), `context` (string)
    - `contextSummary`: string or null
    Generate the JSON schema once at module level: `const CLASSIFIER_JSON_SCHEMA = z.toJSONSchema(TripClassifierSchema)`.
    Derive `type ClassifierOutput = z.infer<typeof TripClassifierSchema>`.

    **Section 4 -- Classifier prompt builder:**
    Private function `buildClassifierPrompt(existingContext, existingDecisions)` returning a string. The system prompt must:
    - State the bot is analyzing WhatsApp group messages for trip planning
    - Mention messages may be Hebrew, English, or mixed
    - Instruct to extract confirmed decisions (not suggestions or questions), open questions, and an updated context summary
    - Include existing trip context and existing decisions in the prompt so Gemini can deduplicate and avoid re-extracting the same decision
    - Instruct to set confidence: "high" for explicit agreement, "medium" for strong implication, "low" for casual mentions (low will be filtered out before DB insert)
    - Instruct not to create a decision if it duplicates an existing one

    **Section 5 -- processTripContext:**
    Private async function `processTripContext(groupJid: string, messages: GroupMsg[]): Promise<void>`. Logic:
    1. Load existing context via `getTripContext(groupJid)` and existing decisions via `getDecisionsByGroup(groupJid)`.
    2. Format messages: join as "senderName: body" lines.
    3. Format existing decisions as a readable list for the prompt (type: value pairs).
    4. Call `generateJson<ClassifierOutput>` with the classifier prompt, formatted messages, CLASSIFIER_JSON_SCHEMA, and schemaName 'trip_context_classifier'.
    5. If result is null, log warning and return.
    6. If contextSummary is non-null, call `upsertTripContext` with destination (from a 'destination' decision if present, otherwise keep existing), dates (same logic), and the new contextSummary.
    7. For each decision in result.decisions: skip if confidence is 'low'. Otherwise call `insertTripDecision` with a crypto.randomUUID() id, groupJid, type, value, confidence, and sourceMessageId set to the first message's id (best-effort attribution).
    8. For each item in result.openItems: call `insertTripDecision` with type 'open_question', value set to the question text, confidence 'high', resolved defaults to false in schema.
    9. Log the number of decisions and open items persisted at info level.

    All error handling: wrap the entire function body in try/catch, log errors, never throw (this runs in a fire-and-forget debounce callback).
